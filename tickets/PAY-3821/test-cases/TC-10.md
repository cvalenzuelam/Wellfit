Validate that restoring SFTP configuration allows idempotent retry of stranded funding batches

**AC Linked:** PAY-3821 / AC-5

**AC Description:** After good config is restored, re-trigger re-attempts pending batches without new retry machinery; synthetic rows prove re-eligibility.

**Preconditions**

- TC-05–TC-09 completed; staged rows still **`RequestSentTimeStamp` NULL**.
- Azure access to restore **`processor.vantiv.batch:sftpurl`** to **`batch.vantivprelive.com`**.

**Expected**

- After restore + re-trigger: staged synthetic rows are **re-selected and re-attempted**.
- With **synthetic** filenames (no real file in storage): rows may remain **NULL** but error shifts from SFTP **connect timeout** to **file not found** (404) — proves retry with good SFTP config.
- **No new** transport-failure timeout alert email for the recovery run (same as TC-06 subject/type).
- Full **stamp** verification requires a **real** batch file (optional extended run — not required for synthetic STAGE pass per `e2e-test-plan.md` TC-04).

**Steps**

First restore **`processor.vantiv.batch:sftpurl`** to **`batch.vantivprelive.com`** in Azure and apply settings.

Then wait ~1 minute if the app restarted.

Next send **POST** `/send-funding-batch` with **`api_key`** and capture response **202**.

Then re-query **`Payments.FundingBatches`** for `PAY3821-E2E-%` and confirm rows still pending (**NULL**) for synthetic data, or stamped if real batch files were used.

Next check App Insights for the recovery run — expect file-not-found or storage error rather than connect timeout.

Finally confirm QA inbox received **no new** PAY-3821 SFTP transport-failure alert for the recovery trigger.
